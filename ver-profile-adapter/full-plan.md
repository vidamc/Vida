# План: профили версий платформы (Ver Profile Adapter)

Документ описывает реализацию **версионируемых профилей Minecraft** в монорепо Vida: одна инфраструктура лоадера, подключаемые артефакты и код «якорей» под конкретную линию игры (**1.21.2–1.21.11**) и поколение платформы (**26.1+**).

---

## 1. Цели и ограничения

### 1.1 Цели

- **Явная граница** между неизменным ядром (discovery, resolver, classloading pipeline, `vida.mod.json`, Latidos/Catalogo/Susurro как концепции) и **профилем платформы** (маппинги, морфы Vifada, `PlatformBridge`, валидация jar).
- **Переключение целевой версии** через один согласованный ключ (`minecraftVersion` / профиль Gradle), без размазанных путей и «магических» констант по репозиторию.
- **Предсказуемый объём работ** при новом game drop: обновление профиля (маппинги + морфы + тесты на замороженный jar), а не переписывание лоадера.
- **Поддержка двух семейств версий**: семантическая линия `1.21.x` и календарная `26.x` с возможным разным **режимом маппингов** (поколение платформы).

### 1.2 Не-цели (честные)

- Автоматический порт на новую версию игры **без** обновления маппингов и таргетов морфов — нереалистично из-за произвольных изменений байткода Mojang.
- Единый «универсальный» публичный API мода, если моддеры протаскивают в стабильные сигнатуры типы `net.minecraft.`* — это отдельная политика API, не решается только профилем.

### 1.3 Принципы терминологии Vida

В коде и документации плана использовать лексику проекта: Cartografía, Vifada, Puertas, Latidos, **без** заимствования чужих имён загрузчиков как шаблона для публичных API.

## 2. Понятия


| Термин                  | Смысл                                                                                                                                                                                     |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Версия игры**         | Строка уровня `1.21.7` или `26.1.2`, идентифицирующая конкретный релиз клиента/сервера.                                                                                                   |
| **Профиль платформы**   | Замкнутый набор: версия игры + хэши/идентификаторы артефактов + дерево маппингов + каталог морфов + реализация платформенного моста + метаданные поколения.                               |
| **Поколение платформы** | Логический ярлык (`LEGACY_121`, `CALENDAR_26`, …), объединяющий правила импорта маппингов, ожидания по обфускации и совместимость JVM. Не обязан совпадать с первой цифрой версии строки. |
| **Якорь**               | Класс/метод ванильного JAR, на который ссылается морф или рефлексия моста; якоря **версионируются** вместе с профилем.                                                                    |

## 3. Архитектура решения

### 3.1 Размещение в монорепо

Рекомендуемая структура (имена каталогов можно уточнить при реализации):

```
ver-profile-adapter/
  full-plan.md                 # этот документ
  README.md                    # краткий вход для разработчиков (после реализации)

platform-profiles/
  README.md                    # как добавить профиль
  schema/                      # JSON Schema или аналог описания профиля
    platform-profile.schema.json

  generations/
    legacy-121/                # поколение 1.21.x (game drops)
      1.21.7/
        profile.toml           # или profile.json — метаданные + ссылки на артефакты
        README.md              # заметки по порту, известные ограничения
      1.21.11/
        ...
    calendar-26/               # поколение 26.x
      26.1/
        ...
```

Артефакты **тяжёлые** (client jar, server jar) в Git не коммитить; хранить:

- URL/SHA-256 в профиле и скачивание в CI через кеш Gradle; или
- локальный путь в `gradle.properties` / переменные окружения для разработчика.

Маппинги (вход Proguard, готовые `.ctg`) — в репозитории **или** в отдельном артефактном модуле, если размер растёт.

### 3.2 Модуль Gradle (опционально, но желательно)

Ввести подпроект, например `:platform-profiles` или расширить `build-logic`:

- задача **валидации профиля**: наличие всех полей, согласованность `generation` и формата версии.
- задача **проверки хэша** jar относительно записи в профиле (fail при несовпадении).
- генерация **ресурса** для рантайма: `META-INF/vida/platform-profile.properties` или один класс-константа с id профиля (для проверки согласованности с реальным jar в рантайме).

### 3.3 Рантайм: выбор профиля

**Источники истины (в порядке приоритета — зафиксировать в коде и документации):**

1. Явная опция агента / `BootOptions`: `minecraftVersion` + опционально `platformProfileId`.
2. Системное свойство `vida.minecraftVersion` / `vida.platformProfile`.
3. Значение, зашитое при сборке лоадера для **каналов** (например fat-jar «только 1.21.7») — только для узких дистрибутивов.

**Сопоставление:** таблица `Version` / строка → запись профиля (classpath resource или embedded manifest). При отсутствии профиля — понятная ошибка на этапе boot, а не «ClassNotFound» глубже по стеку.

### 3.4 Интеграция с существующими модулями


| Модуль                       | Изменения                                                                                                                                                                                            |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **cartografia**              | Импорт маппингов из пути, указанного профилем; поддержка разных **режимов** импорта по `generation` (например различие в namespace или отсутствии обфускации в будущем).                             |
| **loader** / `MappingLoader` | Читать не хардкод путей, а **дескриптор профиля**; передавать в `VidaClassTransformer` контекст профиля.                                                                                             |
| **vifada** / морфы           | Регистрация морфов из **профильного** classpath-модуля (`:platform-morphs-121-7`) или из списка классов в descriptor; исключить смешение морфов разных версий в одном jar лоадера без явного выбора. |
| **dev.vida.platform**        | Интерфейс `PlatformBridge` + **фабрика** `PlatformBridge.load(PlatformProfileContext)`; реализации — в модулях `platform-bridge-121-`*, `platform-bridge-26-`*.                                      |
| **gradle-plugin**            | Расширение `vida.minecraft {}`: поле `platformProfile` или автоматическая привязка `version` → профиль из `platform-profiles/`; синхронизация с `vidaRun`, ремапом.                                  |
| **SyntheticProviders**       | Уже есть версии Vida/Minecraft — убедиться, что синтетический провайдер `minecraft` согласован с выбранным профилем.                                                                                 |


### 3.5 Контракт профиля (минимальные поля)

Зафиксировать в схеме (примерный перечень):

- `gameVersion` — строка.
- `generation` — enum-подобная строка.
- `clientJar` — ожидаемый SHA-256 и опционально имя файла для документации.
- `mappings` — тип входа (`proguard`, `ctg`, …), путь относительно профиля.
- `morphBundle` — список fully-qualified имён классов морфов или ссылка на модуль.
- `platformBridge` — FQCN реализации по умолчанию.
- `minimumJavaVersion` / `recommendedJavaVersion` — для проверки до загрузки игры (опционально).
- `dataPackFormat` / `resourcePackFormat` — если нужны для data-driven прототипа и валидации контента модов.

## 4. Две линии версий: особенности реализации

### 4.1 Линия 1.21.2–1.21.11

- Один **generation** (`legacy-121`), много профилей под каждый дроп.
- Порты между соседними дропами: в основном обновление маппингов и точечная правка якорей морфов и `PlatformBridge`.
- Отдельный документ «матрица поддерживаемых дропов»: какие версии официально поддерживаются CI, какие deprecated.

### 4.2 Линия 26.1+

- Отдельный **generation** (`calendar-26`): свой набор правил парсинга версии (не Confuse с SemVer 1.21.*).
- Возможное расхождение с линией 1.21: JVM, формат jar, степень обфускации — в профиле явные поля `mappingMode`, `artifactLayout`.
- Не смешивать морфы 1.21.11 и 26.1 в одном runtime-профиле без миграции — разные jar, разные якоря.

## 5. Тестирование и CI

### 5.1 Обязательные проверки

- **Контрактный тест на jar**: для каждой поддерживаемой версии — загрузка замороженного client jar (из кеша CI), применение трансформера, минимальный набор assert (класс помечен/метод существует / bridge вызывается в изолированном сценарии).
- **Регрессия маппингов**: снимок дерева имён или хэш нормализованного `.ctg` для детекта случайной подмены файла.
- **Gradle**: задача `verifyPlatformProfiles` на валидность всех профилей в дереве.

### 5.2 Матрица CI

- JDK: минимум версия из профиля поколения (например 21 для 1.21.x, 25 для 26.x когда это станет обязательным).
- OS: хотя бы Linux для каждого профиля; периодически Windows для loader/agent пути.

## 6. Фазы внедрения

### Фаза 0 — Дизайн и схема (1–2 недели)

- Утвердить формат файла профиля (`profile.toml` / JSON) и расположение `platform-profiles/`.
- Описать приоритет источников версии (агент vs Gradle vs embedded).
- Решить политику хранения больших jar (кеш, не Git).

### Фаза 1 — Инфраструктура без полного разнесения морфов (2–4 недели)

- Ввести **идентификатор профиля** и загрузку дескриптора в `BootSequence` / `MappingLoader`.
- Подключить пути маппингов из дескриптора; сохранить обратную совместимость: если профиль не задан — текущее поведение (feature flag или дефолтный профиль `1.21.1`).
- Задачи Gradle: валидация, опционально публикация `META-INF` с id профиля.

### Фаза 2 — Вынос платформенного моста (3–6 недель)

- Разнести реализации `PlatformBridge` по подмодулям per-generation или per-major-drop.
- Все прямые ссылки на ванильные имена классов в лоадере свести к профильным модулям.
- Добавить первый полный профиль для **1.21.1** (baseline) и один дополнительный (**1.21.7** или **1.21.11**) чтобы доказать, что переключение работает end-to-end.

### Фаза 3 — Версионированные морфы (параллельно с фазой 2, по мере готовности)

- Каталог морфов как часть профиля; запрет загрузки морфов «не того» профиля.
- Документация для авторов модов: как объявить совместимость с диапазоном версий игры в `vida.mod.json` (если ещё не полностью есть).

### Фаза 4 — Поколение 26.x (после появления стабильного артефакта и требований)

- Новый `generation` в схеме; отдельный pipeline импорта маппингов при необходимости.
- Профиль `26.1` + контрактные тесты; отдельная job в CI.

### Фаза 5 — Закрепление и документация

- `docs/modules/platform-profiles.md` (или раздел в `loader.md`): как добавить профиль, как прогнать локально, как отладить несовпадение jar.
- Обновление `docs/migration/` при первом breaking изменении в способе указания версии.

## 7. Риски и смягчение


| Риск                                    | Смягчение                                                                                                               |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| Раздувание репозитория jar-ами          | Только хэши + CDN/кеш; локально — переменная `VIDA_MC_CLIENT_JAR`.                                                      |
| Дублирование кода между профилями       | Общий абстрактный слой `platform-common`, профиль только отличия; codegen якорей из маппингов — долгосрочно по желанию. |
| Расхождение версии в лаунчере и профиля | Рантайм-проверка через manifest игры или байт-код маркер + понятное сообщение.                                          |
| Слишком много поддерживаемых дропов     | Политика: N последних дропов линии + LTS-точки; остальное community-maintained профили.                                 |

## 8. Критерии готовности (Definition of Done)

1. Любой разработчик может выбрать профиль в Gradle и получить согласованный ремап/run для **двух** разных версий игры из линии 1.21 без правки Java-кода ядра лоадера.
2. CI выполняет `verifyPlatformProfiles` и хотя бы один контрактный тест трансформации **на каждую** объявленную поддерживаемую версию.
3. Документация описывает поколение `calendar-26` и отличия от `legacy-121`, даже если поддержка 26.x ещё preview.
4. Публичный контракт SemVer для `base`/Gradle не ломается без мажорной версии Vida; изменения только в профилях и опциональных модулях моста.

## 10. Резюме

Реализация **возможна и соответствует** изначальной идее адаптируемости: ядро Vida остаётся общим, а всё, что зависит от конкретного JAR игры, живёт в **версионируемых профилях** и сопутствующих подмодулях моста/морфов. Полный автоматический порт игры без обновления якорей не входит в объём — зато объём правок при каждом дропе становится **ограниченным и локализованным**.
