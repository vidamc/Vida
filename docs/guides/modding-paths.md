# Три дорожки моддинга Vida

Vida сознательно поддерживает **одну экосистему** и **три глубины вмешательства**: от декларативного API до прямого ASM. Этот гайд связывает дорожки в одну карту: когда что выбирать, какие модули читать и в каком порядке осваивать материалы.

## Дорожка A — высокоуровневый API (рекомендуемая по умолчанию)

**Цель:** контент, события, данные, конфиги — без байт-кода и без знания внутренностей Minecraft.

| Тема | Документ |
|------|-----------|
| Точка входа мода, жизненный цикл | [Первый мод](../getting-started/first-mod.md), модуль [base](../modules/base.md) |
| События | [Latidos](./latidos.md) |
| Реестры контента | [Catalogo](./catalogo.md) |
| Настройки | [Ajustes](./ajustes.md) |
| Доступ к полям/методам без mixin | [Puertas](./puertas.md) (access) |

**Gradle:** `dev.vida.mod`, генерация `vida.mod.json`, entrypoints — [gradle-plugin](../modules/gradle-plugin.md).

## Дорожка B — Vifada (структурированная модификация байт-кода)

**Цель:** точечные изменения поведения игры через аннотации и конфигурацию морфов (пакеты сканируются, трансформации согласованы с Cartografía).

| Тема | Документ |
|------|-----------|
| Практика морфов | [Vifada — гайд](./vifada.md) |
| Модуль и термины | [vifada](../modules/vifada.md) |
| Карты имён | [cartografia](../modules/cartografia.md) |

Используйте Vifada, когда нужны хуки в существующие методы классов игры, но не нужен полный контроль над сырым ASM.

---

## Дорожка C — Escultores (низкоуровневые трансформеры)

**Цель:** полный контроль над байт-кодом класса (обход ограничений Vifada, сложные графы инструкций, интеграция со сторонними ASM-пайплайнами).

Объявление классов Escultor в манифесте — см. [manifest-schema](../reference/manifest-schema.md) (`escultores`).

## Как дорожки сочетаются

Один мод может совмещать уровни: например, **API** для регистрации контента + **Vifada** для хука в рендер + **Escultor** для единичного критичного класса. Рекомендуем подниматься по уровню сложности только там, где предыдущая дорожка упирается в ограничения.

Зависимости между модами и политика «что можно загрузить» описываются на этапе резолва — см. [Политики доступа резолвера](./resolver-policies.md).

## Фазы внедрения (для команды и контрибьюторов)

Ниже — **план работ по репозиторию**, а не обучение автора мода. Три фазы из запроса распределены так:

| Фаза | Содержание | Критерий готовности |
|------|------------|---------------------|
| **1** | Единый навигатор по дорожкам A/B/C, перекрёстные ссылки, operational docs | Эта страница + [resolver-policies](./resolver-policies.md) + обновлённые индексы `docs/` |
| **2** | Резолвер: явная политика доступа (`accessDeniedIds`) | `ResolverOptions`, `ResolverError.AccessPolicyDenied`, тесты в `:resolver` |
| **3** | API-модули `entidad`, `mundo`, `render`, `red` выровнены по стилю с `base`/`bloque` | Согласованные `package-info`, статусы `@ApiStatus`, ссылки на Catalogo/Latidos/Tejido |

Фазы **независимы по смыслу** (можно читать доки без релиза резолвера), но **нумерация отражает приоритет документации → инфраструктура → поверхность API**.
