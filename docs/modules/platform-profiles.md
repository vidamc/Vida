# Профили платформы (platform-profiles)

Не отдельный Gradle-модуль, а **канонический набор JSON-дескрипторов** в корне репозитория (`platform-profiles/generations/...`). Они описывают всё, что зависит от конкретного дропа Minecraft и поколения платформы: Cartografía (стратегия загрузки маппингов), перечень платформенных морфов Vifada, класс {@code PlatformBridge}, ожидания по JVM и опционально checksum клиента.

См. также архитектурный план: [`ver-profile-adapter/full-plan.md`](../../ver-profile-adapter/full-plan.md).

## Поколения

| Папка | Поле `generation` | Назначение |
|-------|-------------------|------------|
| `legacy-121/` | `LEGACY_121` | Семантическая линия `1.21.x`, ProGuard-клиентские маппинги из дерева лаунчера по умолчанию. |
| `calendar-26/` | `CALENDAR_26` | Календарная линия `26.x`; правила версий и импорта Cartografía могут отличаться от `1.21.*`. |

Превью-профиль `calendar-26/26.1.preview` существует для CI и документации до появления стабильного артефакта игры.

## Рантайм: приоритет выбора профиля

1. `BootOptions.platformProfileId()`  
2. Системное свойство `-Dvida.platformProfile=…`  
3. (Опционально в будущем) ресурс, зашитый в узкий дистрибутив загрузчика.

Если профиль **не** задан, ядро ведёт себя как раньше: Cartografía читает маппинги только из каталога игры; синтетический провайдер `minecraft` берёт версию из опций загрузчика или из `version.json` на classpath.

Когда профиль задан, синтетический `minecraft` может дополнительно опираться на поле `gameVersion` дескриптора, если лаунчер не передал `-Dvida.minecraftVersion`.

## Gradle (`vida.minecraft`)

- Явно: `platformProfile.set("legacy-121/1.21.7")`  
- По умолчанию: если задана только `version` и она совпадает с `1.21.*` или `26.*`, плагин подставляет convention `legacy-121/<version>` или `calendar-26/<version>` (см. `VidaPlugin.inferPlatformProfileFromGameVersion`).

Задача {@code vidaRun} пробрасывает `-Dvida.minecraftVersion` и `-Dvida.platformProfile`.

## Совместимость морфов модов

Если активен профиль платформы, мод может ограничить свои Vifada-морфы списком профилей:

```json
"custom": {
  "vida": {
    "platformProfileIds": ["legacy-121/1.21.1", "legacy-121/1.21.7"]
  }
}
```

Если список задан и **текущий** `profileId` в него не входит, сканирование морфов из JAR этого мода пропускается (с предупреждением в логе).

## Проверки

Корневая задача `./gradlew verifyPlatformProfiles` валидирует обязательные ключи и согласованность папки `legacy-121` / `calendar-26` с полем `generation`. После проверки она выводит **SHA-256 fingerprint всего набора файлов `profile.json`** (байты файлов в лексикографическом порядке путей): удобно сверять в CI или в заметках к релизу, что дерево профилей совпало с ожидаемым.

## Контрактный список профилей

Файл `platform-profiles/supported-contract-profiles.txt` перечисляет строки `profileId` (по одному на строку; строки с `#` — комментарии). Он копируется в артефакт загрузчика как `META-INF/vida/supported-contract-profiles.txt`. Тест `PlatformProfilesContractTest` параметризован этим списком: для каждой строки загружается профиль и проверяется, что классы из `morphBundle` разрешимы на classpath загрузчика.

Добавляя новый поддерживаемый дроп, обновите профиль JSON, при необходимости этот список и прогоните `./gradlew verifyPlatformProfiles :loader:test`.

## Проверка клиентского JAR (`clientJar.sha256`)

Если в `profile.json` задано поле **`clientJar.sha256`**, при старте выполняется сравнение с файлом клиента:

- путь задаётся системным свойством **`-Dvida.clientJar=…`** (см. javadoc `VidaPremain`), либо, если свойство отсутствует, используется единственный элемент списка **game JAR** из опций загрузчика;
- при несовпадении хэша загрузка завершается ошибкой `BootFailure`;
- если хэш в профиле есть, а путь к JAR определить нельзя — проверка пропускается с предупреждением в логе.

Так можно жёстко привязать профиль к конкретному билду клиента без хранения бинарника в репозитории.
