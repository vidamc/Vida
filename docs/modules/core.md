# core

Базовые примитивы, от которых зависят все остальные модули Vida. Модуль намеренно лёгкий — без сторонних зависимостей — и пригоден для использования на самых ранних фазах запуска.

- Пакет: `dev.vida.core`
- Gradle: `dev.vida:vida-core`
- Стабильность: `@ApiStatus.Stable`

## Состав

### `Identifier`

Namespaced-идентификатор вида `ns:path`. Используется везде, где нужно сослаться на объект в реестре, имя события или пакета.

```java
Identifier id = Identifier.of("miaventura", "espada_sagrada");
// id.namespace() == "miaventura"
// id.path()      == "espada_sagrada"
// id.toString()  == "miaventura:espada_sagrada"
```

Правила:

- `namespace` — строго ASCII, `[a-z0-9_.-]+`;
- `path` — ASCII, `[a-z0-9_./-]+`;
- двоеточие разрешено только как разделитель.

### `Version` / `VersionRange`

SemVer 2.0.0 и NPM-совместимые диапазоны.

```java
Version v = Version.parse("1.2.3-beta.1+build.7");
VersionRange range = VersionRange.parse("^1.0.0"); // 1.x.x
boolean ok = range.contains(v); // true

Version v25 = Version.parse("2.5.0");
boolean still = range.contains(v25); // false
```

Поддерживаемые операторы диапазона: `=`, `!=`, `>`, `>=`, `<`, `<=`, `~`, `^`, `x.x.x`, `1.2.*`, диапазон-through-hyphen `1.0.0 - 1.5.0`, объединение через `||`. Нормализация даёт детерминированное `toString()`.

### `Result<T, E>`

Either-подобный контейнер: либо успех `T`, либо ошибка `E`. Vida использует его для любых операций, где «ошибка — ожидаемая часть контракта» (парсинг, резолвер, чтение кэша). Исключения остаются для непредвиденного.

```java
Result<Version, String> r = Version.tryParse("1.0");
if (r.isOk()) { ... r.value() ... }
else          { ... r.error() ... }

// комбинаторы:
Result<Integer, String> n = r.map(v -> v.major());
```

Контракт: `Result` не является контрол-флоу — его `error()` бросает `IllegalStateException`, если `isOk()`, и наоборот. Используйте `isOk()` / `isErr()` или pattern matching.

### `Log`

Тонкий фасад поверх SLF4J. Две причины существования:

1. Даёт стандартное имя логгера (`vida.<module>` или `vida.mod.<id>`) — инструменты и пользователи знают, что и где фильтровать.
2. Скрывает реализацию: мы можем уйти с SLF4J, не ломая API.

```java
Log log = Log.of("vida.discovery");
log.info("нашли {} модов", count);
log.warn("jar {} без манифеста — пропускаем", path);
```

Уровни: `TRACE`, `DEBUG`, `INFO`, `WARN`, `ERROR`. Формат сообщений — SLF4J placeholder-стиль (`{}`). Исключение добавляется последним аргументом и попадает в stack trace.

### `ApiStatus`

Аннотации-маркеры стабильности публичного API:

```java
@ApiStatus.Stable          // зафиксировано до следующей мажорной версии (для модулей 1.0 — см. api-stability.md)
@ApiStatus.Preview("loader") // пример: рантайм загрузчика ещё в preview
@ApiStatus.Internal        // не использовать извне
```

Аннотации пакетного уровня ставятся в `package-info.java`; классовые — на отдельных классах при необходимости. Полная политика — [reference/api-stability.md](../reference/api-stability.md).

## Почему так мало?

`core` — это словарь. Он попадает на classpath ВСЕГДА: агента, игры, каждого мода. Поэтому мы держим его плоским и без зависимостей. Любое добавление в `core` проходит ревью «нужно ли это всем троим (агент / игра / мод)».

Всё, что можно вынести, вынесено в следующие модули: JSON-парсер — в `manifest`, реестры — в `base`, bytecode — в `vifada` и `cartografia`.

## Совместимость

`@Stable` означает: мы обещаем не ломать публичный контракт между `1.x` и `1.y`. Единственное исключение — добавление новых enum-значений в `ApiStatus`, что формально исходный код не ломает, но может потребовать обновления `switch`-выражений.

Если вам нужен новый примитив в `core`, откройте issue — скорее всего, он принадлежит в другой модуль. Для настоящих примитивов (например, `Result.Partial<T, E, W>` — успех с предупреждениями) мы открыты.

## Тестирование

Тесты в `core/src/test/java` используют JUnit 5 без сторонних либ. `Result` и `Version` — классические примеры для мутационного тестирования (PIT); покрытие ~98%.
