# Архитектура: философия производительности

Эта страница — о решениях, а не об измерениях. Цифры живут в бенчмарках (`bench/`) и меняются между версиями; а вот *принципы*, которые определяют цифры, меняются редко и стоят того, чтобы быть записанными.

## Правило нуля

**После инициализации загрузчик Vida не делает ничего.** Ни тика, ни аллокации, ни системного вызова в `VidaClassTransformer.transform(...)`, который JVM всё равно больше не вызовет.

«Ничего» — это буквально. В рантайм-профиле Vida должен быть статистически неразличим от отсутствующего. Любая работа, оставшаяся у загрузчика после старта, — баг, и мы его фиксим.

Как это достигается:

- Transformer работает только в фазе загрузки классов; `canRetransform = false` запрещает JVM звать его повторно.
- `LongAdder`-счётчики в транзит-коде вместо `AtomicLong` — на случай, если трансформер всё же вызвали (первая загрузка чужого класса); конкурентная запись в `LongAdder` не создаёт контеншн на hot-path.
- Никаких daemon-потоков у агента. Если вам кажется, что Vida должна «что-то мониторить» — это задача API (будущий `Vigia`), а не загрузчика.

## Почему нет `retransform`

Java Agent может декларировать, что умеет переопределять классы на лету (`Can-Retransform-Classes: true`). Это удобно для REPL и некоторых инструментов. Для загрузчика модов — это техдолг:

1. JVM обязана хранить для каждого retransformable-класса оригинальные байты и генерировать side-таблицы, чтобы откатить при необходимости. Лишняя память на каждый класс игры.
2. C2-оптимизатор консервативнее с классами, которые могут быть переопределены: меньше inline, осторожнее спекулятивная девиртуализация.
3. `Instrumentation.retransformClasses(...)` сам по себе — safepoint. Каждая транзакция тормозит главный поток.

Vida всё равно не переопределяет классы в рантайме (модель — «собрать всё на старте, применить при первой загрузке»), так что выигрывать здесь нечего. Выключили — и избавили JVM от обязательств.

### Dev-only hot reload

Перезагрузка классов разрабатываемого мода и сброс реестров **`Catalogo`** включаются **только** через dev-пайплайн Gradle (`hotReload`, см. [guides/hot-reload.md](../guides/hot-reload.md)). Это **не** замена `retransform` для vanilla-классов и **не** часть производственного контракта производительности.

## Почему `BrandingEscultor` работает через byte-scan

Простейший путь изменить format-строку F3 — распарсить класс ASM'ом и пройтись по `ConstantPool`. Это просто, но это дорого для каждого загружаемого класса: мы бы ранили ASM на каждом `transform()`-вызове и выбрасывали результат, если строка не нашлась.

Вместо этого `BrandingEscultor.mightMatch(bytes)` — быстрый byte-scan (ищет ASCII-подстроку в сыром файле) с низким false-positive и нулевым false-negative. ASM запускается только после положительного `mightMatch`. На классах, где форматной строки нет, мы не платим ничего.

Это общий паттерн, который мы рекомендуем всем авторам `Escultor`: дешёвый pre-check, дорогая обработка только при прохождении pre-check'а.

## `Latidos`: без `Method.invoke` в `emitir`

Классическая реализация event-bus'а на Java часто вызывает обработчики через **`Method.invoke`**: boxing, проверки `AccessibleObject`, временные массивы — и C2 почти не инлайнит такой вызов.

В **`DefaultLatidoBus`** при `emitir` шина копирует снимок подписчиков под read-lock и затем вызывает **`Oyente.manejar(...)`** как обычный виртуальный метод — **рефлексии на горячем пути нет**.

**`LatidoRegistrador`** (аннотации `@EjecutorLatido` / `@OyenteDeTick`) один раз сканирует класс рефлексией, затем по возможности строит **`MethodHandle.unreflect` + `bindTo(instance)`** для вызова метода мода; при недоступности `unreflect` остаётся запасной путь `Method.invoke`.

**Платформенный мост** (`VanillaBridge`, `GuiGraphicsFillReflect`): доступ к `Minecraft` / `Window` / `Level` и к `GuiGraphics.fill` — через **закэшированные `MethodHandle`**, чтобы инжекты Vifada в клиентский тик и HUD не тащили `Method.invoke` на каждый тик/кадр.

Подробнее о практике подписок — [guides/latidos.md](../guides/latidos.md).

## Параллельная дискавери

`ModScanner` использует по воркеру на JAR:

- Открытие `ZipFile` и чтение `vida.mod.json` — I/O-bound, отлично распараллеливается.
- Парсинг манифеста — CPU-bound, но дёшев; кладётся на тот же воркер, чтобы не пересылать байты между потоками.
- Вложенные JAR (nested) — рекурсивно, через `BytesZipReader` в памяти.

На SSD с 150 модами на 8-ядерной машине фаза дискавери занимает меньше 300 мс. Bottleneck — не IO, а JDK-шный `ZipFile` с его внутренней блокировкой; поэтому мы держим по одному `ZipFile` на поток (не шарим).

## Кэш разобранных манифестов

`mods.idx` — собственный бинарный формат в `<data>/mods.idx`. Хранит:

- mtime + размер каждого JAR,
- хэш SHA-256 его содержимого,
- разобранный `ModManifest` в sealed-формате.

При следующем старте: для каждого JAR сверяем mtime+size (быстро) → если совпало, сверяем хэш (медленнее, но нужно не всегда) → если совпало, берём манифест из кэша. На «холодном» старте кэш не помогает; на «тёплом» (типичный случай игрока) — срезает до 80% времени дискавери.

Инвалидация:

- изменился сам JAR → автоматически.
- изменилась версия Vida → весь индекс сбрасывается (поле `schema` в заголовке).
- ручной сброс — удалить `<data>/mods.idx`.

## `Path.toRealPath()` и пути на Windows

На Windows `Path.toRealPath()` делает системный вызов для каждого сегмента пути. На ~200 модах это превращалось в секунды. Vida использует `Path.toAbsolutePath().normalize()` в горячем пути и зовёт `toRealPath()` только один раз на весь runtime — чтобы однажды убедиться в canonical-виде корневой папки модов.

## Память

- `ModCandidate` не держит ссылок на байты JAR — только на `ZipReader`, который ленив.
- `MorphIndex` — плотная `HashMap<String, List<MorphSource>>`. На типичном наборе модов занимает десятки КБ.
- Кэш успешных трансформаций байткода (`TransformBytecodeCache`) ограничен по числу записей в памяти: системное свойство **`vida.transformCache.memEntries`** (по умолчанию `512`; LRU), плюс дисковый слой с инвалидацией по хэшам.
- Кэш `tomlj` закрываем через `try-with-resources` сразу после разбора конфига.
- `WeakReference`-кэши для дорогих промежуточных структур (`MappingTree`, парсенный `vida.mod.json` второй раз) — чтобы GC мог освободить их, если на них никто не держит жёстких ссылок.

### RSS и CI

Пик **RSS** на сценариях «0 / много / 300+ модов» фиксируется бенчмарк-харнессом и ручной калибровкой; жёсткий gate по RSS в CI включается только после стабилизации эталона (иначе шум ОС). До этого регрессии по времени резолва покрывает `ResolverLargeUniverseTest` в стандартном `./gradlew test`.

## Что мы сознательно не делаем

- **Не грузим всё лениво.** Есть пределы лени: если бы мы откладывали парсинг манифестов «до первой проверки зависимости», резолвер тоже стал бы ленивым и мы получили бы труднодиагностируемые фейлы в середине старта. Поэтому boot-фаза полностью eager.
- **Не используем off-heap для внутренних структур.** `ByteBuffer` и `Unsafe` красивы в докладах, но добавляют class-of-errors (memory leaks, GC vs native lifetime) без существенного выигрыша на нашей нагрузке.
- **Не делаем кросс-модное AOT.** Идея «собрать один fat-AOT на все моды» красивая, но противоречит дев-сценарию точечной пересборки мода; продакшен остаётся на обычном classpath и агенте без `retransform`.

## Измерения

Бенчмарки живут в `bench/` и используют JMH. Базовые метрики, которые мы гарантируем (на reference-машине, описанной в `bench/README.md`):

- cold start без модов — ≤ 3.5 с от `main(...)` до title screen.
- cold start с 150 модами — ≤ 8 с.
- GC-pressure в rush-phase — < 50 МБ аллокаций за 10 с ingame-геймплея без модов.

Регрессии тех же бенчмарков в CI ломают сборку.
