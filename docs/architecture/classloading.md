# Архитектура: ClassLoader'ы

Три слоя изоляции между агентом, игрой и модами. Эта страница отвечает на вопросы: «почему мод не видит чужие классы», «куда девается `vida-base`» и «как работает `@VifadaShadow`».

## Дерево

```
┌───────────────────────────────────────────────────────────┐
│ Bootstrap / Platform (sun.misc, java.*, jdk.*)            │
└───────────────────────────────────────────────────────────┘
                        ▲
                        │
┌───────────────────────────────────────────────────────────┐
│ System / AppClassLoader                                   │
│   ─ vida-loader-<ver>.jar (агент + зависимости)           │
│   ─ всё, что JVM получила в classpath                     │
└───────────────────────────────────────────────────────────┘
                        ▲
                        │ parent
┌───────────────────────────────────────────────────────────┐
│ JuegoLoader (custom)                                      │
│   ─ net.minecraft.*, com.mojang.*                         │
│   ─ dev.vida.base.*  (публичный API, общий для всех модов)│
│   ─ transformer применяется ЗДЕСЬ                         │
└───────────────────────────────────────────────────────────┘
                        ▲
              ┌─────────┴─────────┐
              │                   │
┌─────────────────────┐  ┌─────────────────────┐
│ ModLoader #1        │  │ ModLoader #2        │  ...
│  ─ классы мода A    │  │  ─ классы мода B    │
└─────────────────────┘  └─────────────────────┘
```

## Почему три слоя

**Agent ↔ Juego.** Vida сама зависит от ASM и SLF4J. Если эти библиотеки окажутся видны игре и модам — моды начнут случайно линковаться с ASM-5.x из Forge-подобной экосистемы, а не с ASM-9.x, который тащит Vida. Разделение на System и Juego гарантирует, что ASM «живёт» только у агента.

**Juego ↔ Mod.** Моды не должны видеть классы друг друга напрямую. Это значит, что два мода с библиотекой `kotlinx.serialization:1.5` и `1.6` не сломают третий мод: каждый пользуется своей версией. Публичный API (`vida-base`) живёт в `JuegoLoader`, потому что он должен быть единственным — все моды пользуются одним `LatidoBus`.

**Mod #1 ↔ Mod #2.** Полная изоляция. Но для публичных API-точек обмена (`Catalogo`, `Latidos`) всё равно работает, потому что сам API загружен выше — в `JuegoLoader`.

## Правила делегирования

Каждый `ModLoader` переопределяет стандартную модель:

```
loadClass(name):
    1. уже загружен? → вернуть
    2. name starts with "java.", "jdk.", "sun.", "dev.vida.base.",
       "net.minecraft.", "com.mojang." → parent.loadClass(name)
    3. класс есть в собственном JAR мода → defineClass из байтов
    4. иначе → parent.loadClass(name)  (пусть решает Juego)
```

Это parent-first **с белым списком**, как обычно и делают изолирующие лоадеры. Важно: mod-to-mod обращения идут только через API (`Catalogo`, `Latidos`), не через прямые `Class.forName` — последнее всё равно пойдёт в parent и не найдёт приватных классов чужого мода.

## Где применяется Vifada

В `JuegoLoader`. Точнее — в `VidaClassTransformer`, зарегистрированном через `Instrumentation.addTransformer(...)`. В момент, когда `JuegoLoader.loadClass("net.minecraft.server.MinecraftServer")` идёт в JVM, она отдаёт сырые байты всем зарегистрированным трансформерам; Vida смотрит в `MorphIndex`, находит список `MorphSource`, применяет их через `dev.vida.vifada.Transformer` и возвращает патченные байты.

`Can-Retransform-Classes: false` означает: мы не патчим классы повторно. Если на момент загрузки `MinecraftServer` морф ещё не собран (например, мод не успел инициализироваться) — он не применится, и JVM запомнит оригинальные байты. Это сильное ограничение, но оно работает, потому что `BootSequence` полностью собирает `MorphIndex` ДО того, как JVM загружает первый класс Minecraft.

## `@VifadaShadow`

Мод пишет в своём морфе:

```java
@VifadaMorph(target = "net.minecraft.world.level.Level")
public abstract class LevelMorph {
    @VifadaShadow private int tickCount;
    @VifadaShadow public abstract void tick();
}
```

Компилятор видит обычные поле и метод; байтовый референс `LevelMorph.tickCount` / `LevelMorph.tick()`. При трансформации Vifada встречает эти референсы в теле inject-методов и подменяет owner на `net.minecraft.world.level.Level` — значит, в рантайме мод _действительно_ обращается к приватному полю/методу vanilla-класса. Всё это — в `JuegoLoader`, так что reflection-не-нужен, и нет cross-loader-барьера.

## `TransformingClassLoader` — альтернативный путь

`VidaBoot.boot(options)` (без агента) создаёт `TransformingClassLoader` и делает его `JuegoLoader`-эквивалентом. Там тот же transformer, но dispatch происходит в `findClass(...)` вместо `ClassFileTransformer`. Это нужно для:

- integration-тестов Vida, где нет живой JVM с `-javaagent`;
- инструментов (анализатор морфов, дампер байткода).

В обычном запуске игры `TransformingClassLoader` не создаётся — всё работает через `Instrumentation`.

## Почему нет `ModuleLayer`

JPMS (`ModuleLayer`) — красивое решение на бумаге, но в 1.21.1 у нас:

- Minecraft распространяется без `module-info.class` — автоматический модуль с именем, выведенным из JAR.
- Многие библиотеки ECO-системы моддинга тоже без JPMS.
- Разные моды могут тащить пересекающиеся пакеты (`shadow`-плагин для Gradle), что при JPMS вызывает `LayerInstantiationException`.

Поэтому в 0.x-релизах Vida остаётся на классической модели лоадеров, оставив JPMS как опцию для 1.x. Решение задокументировано, но не финальное — если сообщество модов массово перейдёт на модули, мы пересмотрим.

## Изоляция ресурсов

`getResource(...)` и `getResources(...)` следуют тем же правилам делегирования, что и классы. Моды видят свои ресурсы и ресурсы `vida-base`/Minecraft; чужие ресурсы — нет. Это важно для:

- `vida.mod.json` — каждый лоадер видит свой и только свой,
- language-файлов и текстур — пока не слиты через `Fuente` API.

## Что это даёт

- Определённость: «я вижу свой код, Minecraft, Vida API и базовые java-классы».
- Возможность грузить библиотеки разных версий в разных модах без коллизий.
- Точную гарантию: после инициализации Vida у `JuegoLoader` фиксированное множество трансформеров и байтов; новые классы не появляются (см. [performance.md](./performance.md)).
